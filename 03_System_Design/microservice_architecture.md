# Delivery App — микросервисная архитектура

## 1. Цель архитектуры
Разделить систему доставки еды на независимые сервисы, каждый из которых отвечает за отдельную бизнес-область: заказы, рестораны, курьеры, платежи, уведомления и пользователей.

Архитектура должна обеспечить:
- независимое развитие сервисов
- масштабирование наиболее нагруженных компонентов
- снижение связности между модулями
- асинхронную обработку событий жизненного цикла заказа

## 2. Сервисы

### API Gateway
Ответственность:
- Прием запросов от приложений
- Маршрутизация запросов
- Авторизация
- Rate limiting
- Единый вход в систему

### Order Service
Ответственность:
- Подтверждение заказа
- Изменение статуса заказа
- Публикацию order_confirmed
- Связь заказа с клиентом, рестораном, курьером

### Restaurant Service
Ответственность:
- Рестораны
- Меню
- Доступность ресторана
- Доступность позиций меню

### Courier Service
Ответственность:
- Курьеры
- Статус курьеров
- Поиск доступного курьера
- Назначение курьера

### Payment Service
Ответственность:
- Создание платежа
- Подтверждение оплаты
- Статус оплаты
- Возврат денежных средств

### Notification Service
Ответственность:
- Push-уведомления
- Email и SMS
- Уведомления о статусах заказа

### User Service
Ответственность:
- Пользователи
- Роли
- Профили
- Адрес доставки

### Message Broker
Ответственность:
- Передача событий между сервисами
- Асинхронная обработка сообщений
- Снижение прямой связности между сервисами

## 3. Типы взаимодействий
### REST
- Используется для синхронных операций, когда вызывающей стороне нужен ответ сразу.

#### Примеры:
- Client App → API Gateway → Order Service: создать заказ
- Restaurant App → API Gateway → Order Service: подтвердить заказ
- Courier App → API Gateway → Courier Service: принять заказ

### Events
- Используются для уведомления других сервисов о произошедших событиях без прямой зависимости.

#### Поток событий:
- Order Service → Message Broker → Restaurant Service
- Order Service → Message Broker → Notification Service
- Order Service → Message Broker → Courier Service

#### Примеры:
- Order Service публикует order_created
- Restaurant Service обрабатывает order_created
- Notification Service отправляет уведомление клиенту
- Courier Service запускает поиск курьера после order_confirmed

## 4. События
- order_created
- order_confirmed
- order_canceled
- courier_assigned
- order_delivered
- payment_created
- payment_confirmed

## 5. Владение данными
- User Service владеет данными пользователей и адресами доставки
- Restaurant Service владеет ресторанами и меню
- Order Service владеет заказами и статусами заказов
- Courier Service владеет курьерами и их доступностью
- Payment Service владеет платежами
- Notification Service владеет шаблонами и статусами уведомлений

## 6. Базы данных
Каждый сервис имеет собственную базу данных. Прямой доступ одного сервиса к базе данных другого сервиса запрещен.

Если сервису нужны данные другого домена, он получает их через API или события.

## 7. Основной поток создания заказа
1. Client App отправляет запрос в API Gateway
2. API Gateway передает запрос в Order Service
3. Order Service запрашивает у Restaurant Service доступность ресторана и позиций меню
4. Order Service создает заказ со статусом created
5. Order Service публикует событие order_created
6. Restaurant Service получает событие order_created и отображает заказ ресторану
7. Notification Service получает событие order_created и уведомляет клиента

## 8. Архитектурные ограничения
- Сервисы не обращаются напрямую к базам данных друг друга
- Синхронные вызовы используются только там, где нужен немедленный ответ
- События используются для уведомления о завершенных бизнес-фактах
- Order Service является владельцем жизненного цикла заказа
