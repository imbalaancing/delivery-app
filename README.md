# Delivery App — System Analysis Case

Учебный проект по системному анализу: проектирование MVP сервиса доставки еды от бизнес-контекста и требований до API, модели данных и базового архитектурного представления.

Проект показывает полный цикл аналитической проработки системы: от бизнес-задачи и пользовательских сценариев до диаграмм, REST API, OpenAPI-контракта, SQL-схемы, тест-кейсов и базовой микросервисной декомпозиции.

---

## 1. Цель проекта

Цель проекта — продемонстрировать навыки системного аналитика на примере сервиса доставки еды.

В рамках проекта выполнены:

- анализ бизнес-контекста;
- определение MVP;
- описание пользователей и пользовательских сценариев;
- формирование User Stories и Acceptance Criteria;
- моделирование бизнес-процесса;
- описание жизненного цикла заказа;
- проектирование взаимодействия сервисов;
- проектирование модели данных;
- описание REST API;
- оформление OpenAPI-контракта;
- подготовка API test cases;
- базовая декомпозиция системы на сервисы;
- описание событийной модели.

---

## 2. Описание системы

**Delivery App** — сервис заказа и доставки еды, который обеспечивает взаимодействие между клиентами, ресторанами и курьерами.

Основной сценарий системы:

1. Клиент выбирает ресторан и блюда.
2. Клиент оформляет заказ.
3. Ресторан получает и подтверждает заказ.
4. Система запускает поиск курьера.
5. Курьер принимает заказ.
6. Заказ передается курьеру.
7. Курьер доставляет заказ клиенту.
8. Клиент и участники процесса получают уведомления о статусах заказа.

---

## 3. Пользователи системы

- **Клиент** — оформляет заказ, выбирает ресторан и блюда, отслеживает статус доставки.
- **Ресторан** — получает заказ, подтверждает его и готовит к выдаче.
- **Курьер** — принимает заказ и доставляет его клиенту.
- **Поддержка** — просматривает заказы и обрабатывает спорные ситуации.

---

## 4. MVP-функциональность

В рамках MVP рассмотрены основные сценарии:

- оформление заказа клиентом;
- подтверждение заказа рестораном;
- автоматическое назначение курьера;
- принятие заказа курьером;
- доставка заказа клиенту;
- отслеживание статуса заказа;
- отмена заказа;
- обработка нештатных ситуаций.

---

## 5. Основной бизнес-процесс

Основной процесс оформления и доставки заказа:

1. Клиент оформляет заказ в приложении.
2. Order Service создает заказ.
3. Restaurant Service передает заказ ресторану.
4. Ресторан подтверждает заказ.
5. Courier Service запускает поиск доступного курьера.
6. Курьер принимает заказ.
7. Заказ получает статус `assigned_to_courier`.
8. Курьер доставляет заказ клиенту.
9. Заказ получает статус `delivered`.

Также предусмотрены альтернативные сценарии:

- ресторан отклоняет заказ;
- курьер не найден;
- курьер отклоняет назначение;
- заказ отменяется;
- происходит ошибка оплаты или уведомления.

## 6. Жизненный цикл заказа

В проекте используется жизненный цикл заказа:

```text
created
  ↓
confirmed_by_restaurant
  ↓
cooking
  ↓
ready_for_pickup
  ↓
assigned_to_courier
  ↓
delivering
  ↓
delivered
```

Также предусмотрен финальный статус:

```text
canceled
```

## 7. API

В проекте описан REST API для MVP Delivery App.

Ключевые endpoint’ы:

- `POST /orders` — создание заказа;
- `GET /orders/{orderId}` — получение информации о заказе;
- `POST /orders/{orderId}/confirm` — подтверждение заказа рестораном;
- `POST /orders/{orderId}/courier/accept` — принятие заказа курьером.

Для endpoint’ов описаны:

- request body;
- path parameters;
- response models;
- HTTP status codes;
- error responses;
- примеры запросов и ответов;
- схемы данных в OpenAPI.

## 8. Модель данных

В проекте спроектирована логическая модель данных и ER-диаграмма.

Ключевые сущности:

- `Client`
- `Restaurant`
- `Courier`
- `DeliveryAddress`
- `MenuItem`
- `Order`
- `OrderItem`
- `Payment`

Ключевые решения по модели данных:

- заказ связан с клиентом, рестораном, адресом доставки и курьером;
- один заказ содержит одну или несколько позиций заказа;
- позиции заказа хранят состав заказа и цену на момент оформления;
- платеж вынесен в отдельную сущность;
- статусы заказа отражают бизнес-состояние доставки.

---

## 9. Архитектурный подход

Для проекта выполнена базовая декомпозиция на сервисы:

- API Gateway;
- Order Service;
- Restaurant Service;
- Courier Service;
- Payment Service;
- Notification Service;
- User Service;
- Message Broker.

Основные архитектурные решения:

- клиентские приложения взаимодействуют с backend через API Gateway;
- Order Service является владельцем жизненного цикла заказа;
- каждый сервис отвечает за свою область данных;
- прямой доступ сервисов к базам данных других сервисов запрещен;
- REST используется для синхронных операций, где нужен немедленный ответ;
- события используются для уведомления других сервисов о произошедших бизнес-фактах;
- асинхронные события передаются через Message Broker;
- Notification Service не меняет статус заказа, а только отправляет уведомления.
