# Delivery App — System Analysis Case

Учебный проект по системному анализу: проектирование сервиса доставки еды от бизнес-контекста и требований до API, модели данных и базового архитектурного представления.

## 1. Цель проекта

Цель проекта — показать полный цикл работы системного аналитика на примере сервиса доставки еды:

- анализ бизнес-контекста;
- формирование MVP;
- описание пользовательских сценариев;
- моделирование бизнес-процесса;
- проектирование взаимодействий между системами;
- проектирование модели данных;
- описание REST API;
- подготовка тест-кейсов;
- базовая декомпозиция на микросервисы;
- описание событийной модели.

Проект демонстрирует, как бизнес-задача постепенно раскладывается на требования, процессы, данные, API-контракты и архитектурные решения.

---

## 2. Описание системы

Delivery App — сервис заказа и доставки еды, который обеспечивает взаимодействие между клиентами, ресторанами и курьерами.

Основные пользователи системы:

- клиент — оформляет заказ, выбирает ресторан и блюда, отслеживает статус доставки;
- ресторан — получает заказ, подтверждает его и готовит к выдаче;
- курьер — принимает заказ и доставляет его клиенту;
- поддержка — просматривает заказы и обрабатывает спорные ситуации.

---

## 3. MVP-функциональность

В рамках MVP рассмотрены основные сценарии:

1. оформление заказа клиентом;
2. подтверждение заказа рестораном;
3. автоматическое назначение курьера;
4. доставка заказа;
5. отслеживание статуса заказа;
6. отмена заказа и обработка нештатных ситуаций.

## 5. Артефакты проекта

### 5.1 Business Context

Описан бизнес-контекст системы:

- цель системы;
- пользователи;
- AS-IS процесс;
- TO-BE процесс;
- ключевые сценарии;
- основные сущности;
- статусы заказа;
- нештатные ситуации.

### 5.2 User Stories и Acceptance Criteria

Для MVP подготовлены пользовательские истории по основным сценариям:

- оформление заказа клиентом;
- подтверждение заказа рестораном;
- назначение курьера;
- доставка заказа;
- отслеживание статуса.

Для историй сформированы критерии приемки и сценарии проверки в формате Given / When / Then.

### 5.3 BPMN

Построена BPMN-диаграмма процесса оформления и доставки заказа.

Диаграмма отражает:

- роли участников процесса;
- создание заказа;
- подтверждение рестораном;
- поиск курьера;
- передачу заказа курьеру;
- доставку клиенту;
- отмену заказа;
- таймауты и альтернативные ветки.

### 5.4 Sequence Diagram

Построена UML Sequence Diagram для процесса оформления заказа и назначения курьера.

Диаграмма показывает взаимодействие между:

- Client App;
- Backend;
- Order Service;
- Restaurant Service;
- Courier Service;
- Courier App.

### 5.5 ERD и модель данных

Спроектирована логическая модель данных и ER-диаграмма.

Ключевые сущности:

- Client;
- Restaurant;
- Courier;
- DeliveryAddress;
- MenuItem;
- Order;
- OrderItem;
- Payment.

Для сущностей выделены основные поля, связи, первичные и внешние ключи.

### 5.6 SQL schema

Подготовлен SQL-скрипт создания таблиц для PostgreSQL-стиля хранения данных.

В схеме отражены:

- таблицы клиентов, ресторанов, курьеров;
- таблицы заказов и позиций заказа;
- таблица адресов доставки;
- таблица платежей;
- связи через foreign keys;
- ограничение `UNIQUE` для связи заказа и платежа в рамках MVP.

### 5.7 OpenAPI

Подготовлен OpenAPI 3.0.3 контракт для MVP API.

Описаны endpoint'ы:

- `POST /orders` — создание заказа;
- `GET /orders/{orderId}` — получение информации о заказе;
- `POST /orders/{orderId}/confirm` — подтверждение заказа рестораном;
- `POST /orders/{orderId}/courier/accept` — принятие заказа курьером.

Для endpoint'ов описаны:

- request body;
- path parameters;
- response models;
- коды ошибок;
- схемы данных;
- примеры запросов и ответов.

### 5.8 API test cases

Подготовлены тест-кейсы для проверки API:

- успешное создание заказа;
- ошибка при некорректных данных заказа;
- успешное подтверждение заказа;
- ошибка доступа при подтверждении заказа;
- успешное назначение курьера;
- ошибка при некорректном формате данных;
- конфликт при повторном назначении курьера.

### 5.9 Microservice Architecture

Система декомпозирована на сервисы:

- API Gateway;
- Order Service;
- Restaurant Service;
- Courier Service;
- Payment Service;
- Notification Service;
- User Service;
- Message Broker.

Зафиксированы зоны ответственности сервисов, типы взаимодействий, владение данными и архитектурные ограничения.

### 5.10 C4 Context и C4 Container

Подготовлены C4-диаграммы:

- System Context Diagram — показывает систему снаружи, пользователей и внешние системы;
- Container Diagram — показывает приложения, backend-сервисы, базы данных, message broker и внешние провайдеры.

### 5.11 Event Matrix

Подготовлена матрица событий для event-driven взаимодействий.

Описаны события:

- `order_created`;
- `order_confirmed`;
- `payment_confirmed`;
- `courier_assigned`;
- `order_canceled`;
- `order_delivered`.

Для событий описаны publisher, topic, trigger, consumers, действия consumers, payload и правила обработки.

## 6. Архитектурные решения

В проекте зафиксированы следующие решения:

- клиентские приложения взаимодействуют с backend через API Gateway;
- Order Service является владельцем жизненного цикла заказа;
- каждый сервис владеет собственной областью данных;
- прямой доступ сервисов к базам данных других сервисов запрещен;
- REST используется для синхронных операций, где нужен немедленный ответ;
- события используются для уведомления других сервисов о произошедших бизнес-фактах;
- асинхронные события передаются через Message Broker;
- Notification Service не меняет статус заказа, а только отправляет уведомления.
